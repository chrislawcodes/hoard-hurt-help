# Reuse Audit — mcp-oauth (no-duplication scan)

Read-only scan against `feat/mcp-oauth`. For each capability the feature needs, this
maps what already exists, then verdicts it as **reuse** (call it as-is),
**extend** (add to/wrap an existing thing), or **justified-new** (nothing fits and
here's why). The bias is toward reuse/extend; `justified-new` must earn it.

## Gaps / nothing exists yet (justified-new)

These have **no existing analog** in the repo. Each is genuinely new, with the reason
nothing fits.

| capability | nearest existing thing | verdict | why nothing fits |
|---|---|---|---|
| OAuth **Resource Server**: 401 + `WWW-Authenticate: Bearer` on `/mcp` (FR-001) | — we are an OAuth *client* only (`app/auth/google.py`) | justified-new | Nothing in the repo acts as a resource server or emits `WWW-Authenticate`. Human login is a server-side session cookie, not a bearer-token gate. Supplied by standalone `fastmcp`'s `GoogleProvider`/`OAuthProxy` (Decision 1), not hand-rolled. |
| RFC 9728 **Protected Resource Metadata** + RFC 8414 / OIDC **AS metadata** + DCR shim + PKCE + audience binding (FR-002/003/004) | — | justified-new | No `/.well-known/*` route, no AS metadata, no DCR, no PKCE, no audience check anywhere. (`grep` for `.well-known` finds only Google's discovery URL in `app/auth/google.py:11`, which we *consume*.) All provided by `fastmcp` v3 out of the box per Decision 1 — the plan must mount these routes relative to the `/mcp` mount (Open design point 4) but they are not reimplemented in our code. |
| **OAuth-identity → canonical Mode A `Connection`** bridge layer (FR-006) | `app/deps.py:188-228` materializes a `Connection` from a *pending `ConnectionSetup`*; `connections.user_id` is **indexed, NOT unique** (`connection.py:37-39`) | justified-new | The only existing "create a Connection" path is the connector's pending-setup materialization, keyed by a `sk_conn_` the user pasted — there is no "one canonical per-user connection" concept and no per-user uniqueness. FR-006 explicitly calls the in-app find-or-create race-prone and demands a DB uniqueness mechanism (a per-user Mode A marker + transactional upsert/lock). That marker + constraint + idempotent bridge does not exist. This is the load-bearing new module. |
| **Encryption-at-rest** helper (FR-012 candidate 1a) | none — `grep -i fernet\|cryptography\|encrypt` over `app/`, `mcp_server/`, `pyproject.toml` returns **zero hits**; the repo stores only sha256 hashes | justified-new | There is no symmetric-encryption utility and `cryptography`/`fernet` is not even a dependency. If the plan picks "store the Mode A raw key encrypted at rest," that helper + a new column + a key-management decision are all new. (Reuse path exists if the plan instead picks FR-012 candidate 1b "short-lived internal token" or 1c "in-process service call" — both lean on existing `tokens.py`/`require_connection` rather than new crypto.) |
| **JWT signing key** lifecycle/storage (FR-013, Open design point 5) | `session_secret` in `app/config.py:39-40` (session-cookie signing only) | justified-new | We own a JWT signing key for the first time (the MCP server mints tokens for clients). `session_secret` is a different secret with a different blast radius; reusing it would conflate session forgery with MCP-token forgery. New config field, fail-loud per FR-013. |
| **Public-tool carve-out under the OAuth gate** (FR-008, Open design point 7) | `get_game_state` is public today only because the whole server is unauthenticated (`mcp_server/server.py:273-292`) | justified-new | An `OAuthProxy`/`GoogleProvider` gates the *whole* `/mcp` app, so the current "no `ctx`, no auth" tool would be accidentally gated. Keeping it public (separate unprotected mount/carve-out) or consciously making it auth-required — plus a regression test — is new behavior the framework does not give for free. |

## Notes for the plan

- **The credential mechanism (FR-012) decides how much is new.** Candidate 1c
  (in-process service call: resolve the `Connection` directly in the MCP tool, skip
  the HTTP loopback) maximizes reuse — it leans on `require_connection`'s logic
  without minting/storing any new secret — but the discovery non-goal currently
  defers it. Candidate 1a (encrypt-at-rest) forces a new crypto dependency and a
  schema change. Candidate 1b (short-lived internal token) is the middle path.
- **Per-user uniqueness is the real new schema work**, not the connection itself.
  `connections.user_id` is only indexed; FR-006 wants a DB-enforced "one Mode A
  connection per user." Likely a partial unique index / unique marker column +
  transactional upsert. Tests must exercise concurrent OAuth callbacks.
