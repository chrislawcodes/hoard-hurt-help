## Architecture decisions (driven by reuse-report.md + plan review)

1. **New `model_verifications` store (justified-new).** Table keyed `(connection_id, provider, model)`: `status` (`unknown`/`checking`/`verified`/`failed`/`timeout`), `error_text` (≤300, sanitized), `consecutive_timeouts` int, `checked_at`. (`connection_providers` is unique per `(connection, provider)` and can't hold per-model rows.) New `app/models/model_verification.py`; migration `0045` — `create_table` upgrade **and a `drop_table` downgrade** (the round-trip test runs `downgrade base`).
2. **Two dedicated endpoints (justified-new), shaped like `report_pid`**, in new `app/routes/agent_model_verification.py` (its own `APIRouter(prefix="/api/agent")`, **imported + `include_router`'d in `app/main.py` with no second prefix** — mirror the `agent_next_turn.router` registration, not the page-router `prefix=` pattern):
   - **GET `/api/agent/model-worklist`** (down): the `(provider, model)` set to verify for this connection. **For every enabled, non-empty-allowlist provider it includes the provider DEFAULT (`default_model_for_provider`) — this is the model most seats actually run (preferred_model is NULL for nearly all agents), and the HIGH finding was that omitting it leaves the real model unverified — PLUS the distinct non-NULL `Agent.preferred_model` values for that provider across the user's agents, PLUS any still-referenced cached row older than the 6h refresh.** Excludes empty-allowlist providers.
   - **POST `/api/agent/model-verification`** (up): list of `{provider, model, outcome, error_text}` upserts (server sanitizes); and the play-time failure `{match_id, provider, model, reason}` (FR-009).
3. **Server logic** in new `app/engine/model_verification.py` (mirrors `provider_readiness`/`connection_health`): `compute_worklist(db, connection)`, `record_results(db, connection, results)` (applies the FR-009a outcome→status mapping incl. the FR-013 `consecutive_timeouts`→`failed` escalation), `model_status_for(db, user, provider, model)` (union over the user's **live machine** connections — reuse the existing `provider_readiness`/`connection_health` liveness predicates, do NOT hand-roll; any `verified` row ⇒ runnable; exclude MCP/paused), and `sanitize_error(text)`.
4. **Connector verification side-task (extend the clock-gate, fix the idle gap).** Add `_VERIFY_INTERVAL = 60` and a **pure, unit-testable predicate `_should_verify(now, last_verify)`** (like `_poll_failed`). **Cap the idle sleep at `min(next_poll_after_seconds, _VERIFY_INTERVAL)`** so the tick fires during far-from-game idle (the loop otherwise sleeps up to 300s). Each tick: GET worklist → per `(provider, model)` run a cheap isolated test call (`<cli> --model <m> --print "ok"` style, ~30s timeout, via a dedicated helper that reuses `_run`'s DEVNULL/cwd/capture but NOT the `_MAX_CONCURRENCY` turn executor) → classify → best-effort POST (try/except `httpx.HTTPError`, like `_report_pid`).
5. **Shared classifier (one source of truth).** A single pure mapping `(exit_code, stderr) → outcome` used by BOTH the connector test call and the play-time path; a parametrized table test exercises it. FR-005 success predicate (exit 0 + non-empty = `verified`); FR-009a (model-unavailable patterns → `failed`; timeout/PATH/network/unclassifiable → `timeout`). Synthetic stderr fixtures (Gemini sample is synthetic — CLI is dead here).
6. **Per-model status UI (mirror ProviderReadiness, new axis).** `model_status_for` → status enum → badge map → HTMX fragment on agent-settings. **The badge's provider is derived from the model via `provider_for_model(model)`** (the agent isn't pinned to a provider); when `preferred_model` is NULL the page shows the per-provider default's status for the providers the user has connected.
7. **Settings picker (surface over the shipped column).** Agent settings/detail route + template: advanced "Preferred model" `<select>` (options from `PROVIDER_MODELS`, grouped) → writes `Agent.preferred_model`; plus the status badge and read-only effective-model line (FR-010; empty-allowlist machine seat shows "runs your <provider> config's model"). Join page untouched (FR-011).
8. **Join warning (FR-014)** in the join flow: `model_status_for` union; warn (not block) only when verified-failing on every live machine connection for that provider; silent on unknown/checking.

## Residual Risks

- **Connector idle cadence.** Capped idle sleep + `_should_verify` predicate. verification: unit-test `_should_verify` returns False before 60s and True at/after, like `test_poll_failed_*` — no live server needed.
- **Classifier brittleness across CLI versions / dead Gemini.** verification: parametrized table test over committed **synthetic** stderr samples (claude 404, codex unknown-model, gemini synthetic) → each maps to `failed`; `TimeoutExpired`/`FileNotFoundError` → `timeout`. Both connector and server run the same table.
- **Worklist growth / deprecated models (FR-017).** verification: test `compute_worklist` returns only currently-referenced models (live agents' preferred + provider defaults) + still-referenced stale rows — a removed/deprecated model is NOT re-queued; cap + log if truncated.
- **Empty-allowlist machine seat display.** verification: route test renders a hermes machine seat's effective-model line and asserts "runs your hermes config's model" (not blank / not a fake id).
- **Cross-process classifier divergence.** verification: the single parametrized `(exit,stderr)→status` table is imported and asserted by both the connector test and the server `record_results` test, so the two can't silently diverge.
- **Multi-machine last-writer-wins (Open decision 3, accepted).** verification: `model_status_for` test — one failing machine plus one verified machine ⇒ union still runnable (no false join warning).
