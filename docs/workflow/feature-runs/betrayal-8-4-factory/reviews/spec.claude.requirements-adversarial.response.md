## Findings

### HIGH — Scope omits two UI files that hardcode the stale "-8 if betraying" string [CODE-CONFIRMED]

AC5 requires "No stale −8 anywhere in the UI (chip, caption, groups)," AC6 requires "No stale `BETRAYAL_HURT_POINTS` references remain anywhere in the repo," and R2's verification greps for a literal `-8`/`−8` in "the game module + design doc." But two human-facing UI templates hardcode the exact string `-4 to another, -8 if betraying`:

- `app/templates/fragments/move_legend.html:5`
- `app/templates/fragments/robot_circle/_markup.html:61`

Under the 8/4 re-split the victim's damage is now −4, so "-8 if betraying" becomes flatly false text visible on the home page and the robot-circle stage. Neither file is in the spec's §4 "In scope" list (which names only `viewer.py`, `game.py`, `scoring.py`, `rules.py`, tests, and docs). R2's grep guidance is also scoped too narrowly — it looks in "the game module + design doc," not `app/templates/`, so the letter of the spec's own verification would pass while the stale −8 survives in the UI. A builder following the scope exactly satisfies the spec but misses the intent (AC5/AC6). Fix: add both template files to scope and broaden R2's grep to `app/templates/`.

### HIGH — Robot-circle animation never surfaces the attacker's +4 bonus, so AC5 "attacker +4" is unmet in the primary viewer [CODE-CONFIRMED]

§3.4 and AC5 require the viewer to show "attacker +4" honestly. The spec's §3.4 only addresses `build_pd_replay_view` (the feed chip's `display_delta`). But the flagship viewer surface is the animated robot circle in `_replay_script.html`, and its per-move delta labels are hardcoded literals independent of the server payload:

- HURT: `showDelta(T, -4)` (line 915) — target only; no attacker delta shown for a betrayal.
- The attacker's action in a betraying HURT gets no `showDelta(el, +4)` at all.

So even after §3.4 is implemented, the animation will show the victim −4 (which now happens to be correct) but will never show the attacker's +4 gain — exactly the "who gains" honesty AC5 demands. The spec's acceptance criteria and viewer-honesty section do not mention `_replay_script.html`, and it is not in scope. This is under-specified acceptance: AC5 is written as if `display_delta` reaches the whole UI, but the animation does not read it.

### MEDIUM — "victim = −4" acceptance is ambiguous: raw delta vs. floored delta [CODE-CONFIRMED]

AC1 states "victim = **−4**" and §8 lists a test for "betrayal + floor (victim near 0)." But AC1/§2 describe the *raw* per-turn attribution, while `resolve_turn` floors the **summed** per-player delta and persists `points_delta = new_score - current_round_score` (scoring.py:316-321). A victim starting at 2 who is betrayed ends at 0 with a persisted `points_delta` of −2, not −4. The spec never states which number the AC1 resolver test must assert (the raw −4 attribution, or the post-floor persisted delta). The existing `test_betraying_a_helper_hurts_for_eight` sidesteps this by seeding the victim at 10 so no floor triggers; the new test should do the same and AC1 should say "raw victim damage −4 (pre-floor), floor applies to the summed delta as usual." As written a builder could assert either and claim conformance.

### MEDIUM — `apply_inround_turn` attacker-credit is under-specified; the resolver's HELP-branch equivalence is not stated [CODE-CONFIRMED]

§3.3 says the mirror's betrayal handling is "the victim loses `HURT_POINTS` (−4, floored at 0) and the attacker gains `BETRAYAL_BONUS` (+4)." That yields attacker +4 from the bonus, but AC1 claims the attacker's *turn total* is +8. In the resolver, the other +4 comes from the ordinary HELP branch crediting the attacker (scoring.py:285-286), which runs independently. In `apply_inround_turn`, the HELP the victim casts is a separate `HELP` action dict; §3.3 does not state that the mirror must ALSO credit that +4 through its own HELP branch for the attacker's mirror total to reach +8. The mirror currently credits a non-mutual HELP to the *target* (scoring.py:363), so the victim's HELP-to-attacker already credits the attacker +4 there — but the spec does not call this out, and R1's mirror test only asserts "victim −4, attacker +4" (the bonus alone), not the +8 total. So R1's verification does not actually prove the mirror agrees with the resolver's attacker +8; it only checks the +4 bonus in isolation. Acceptance criterion for the mirror should assert the full attacker +8, matching AC1.

### MEDIUM — Constant rename will break `tests/test_rules_text.py` and `tests/test_inround_mirror.py` imports, but §4 doesn't flag the rename fan-out [CODE-CONFIRMED]

§3.1 removes `BETRAYAL_HURT_POINTS` and adds `BETRAYAL_BONUS`. Per the spec-writing rule "list old name → new name explicitly" and "removed/renamed symbols." §4 lists the test files to edit but does not enumerate the concrete symbol breakages a rename causes:

- `tests/test_rules_text.py:8,19,20` imports `BETRAYAL_HURT_POINTS` and asserts `f"-{BETRAYAL_HURT_POINTS}" in GAME_RULES_TEXT` and `BETRAYAL_HURT_POINTS != HURT_POINTS`. After the rename the import is an `ImportError` (collection failure), and the `!= HURT_POINTS` assertion is meaningless for a +4 bonus that equals `HURT_POINTS`.
- `tests/test_inround_mirror.py:5` references `BETRAYAL_HURT_POINTS` in its docstring (cosmetic, but stale).
- `app/games/hoard_hurt_help/viewer.py:23,396` and `scoring.py:17,178,181,227,255` all import/use it — §4 covers scoring.py and viewer.py generically but does not say "the rename forces edits at these exact call sites, and `test_rules_text` must switch to asserting the new +4/−4 wording." A builder who edits only the resolver could leave the mirror or test imports broken. AC7 (preflight green) would catch it, but the spec should name the rename fan-out so it isn't discovered only at the gate.

### LOW — `match_summary.py` reads `display_delta` but the spec's non-goals never confirm it's unaffected [CODE-CONFIRMED]

`match_summary.py:84` consumes `action.get("display_delta")`. The spec changes the attacker's `display_delta` (adding +4 in `build_pd_replay_view`) and drops the victim's −8 override. `build_final_summary` only looks at positive deltas (biggest-gift/pact framing), so surfacing an attacker +4 could newly make a betraying attacker the "biggest gift" giver in a finale summary — a subtle intent question. The old impact-review flagged this as Tier 5 "won't break, may want follow-up," but the current spec's non-goals don't mention `match_summary.py` at all, leaving it unstated whether the finale summary's behavior is allowed to shift. Low because it won't crash and the effect is cosmetic, but it's an unlisted downstream reader of a field the spec changes.

### LOW — AC6 "anywhere in the repo" contradicts §4.3's reality-check about non-existent files, and §4's scope list is narrower than "anywhere" [CODE-CONFIRMED]

AC6 demands "No stale `BETRAYAL_HURT_POINTS` references remain **anywhere in the repo**." Confirmed remaining references live in `docs/games/hoard-hurt-help/betray-helper-impact-review.md:39,43` (which §4 does list, as "mark superseded/implemented" — but "mark superseded" is not the same as "remove the constant name," so a builder could leave the string while calling the doc superseded and still fail AC6's literal wording). Also, the `feature-runs/` history and this very spec contain the token; AC6 should scope itself to shipping code + design docs, matching §4, rather than "anywhere in the repo," which is self-contradicting given the run artifacts.

## Residual Risks

- **JS running-score mirror divergence (pre-existing, widened in relevance).** `_replay_script.html:100` hardcodes mutual HELP as `+8` (already stale under mutual-help-decay) and HURT as −4 with no betrayal awareness. This spec makes the victim −4 correct there by luck, but the client sim still won't reflect the attacker's +4, so the animated rail score can disagree with the server. Out of the spec's stated scope; flagged so it isn't assumed fixed.
- **`viewer_headline.py` betrayal beat priority reads `display_delta`.** The cross-turn `betrayal` beat weights by `100 + abs(display_delta)` (line 108) and the "swing" beat filters HURTs by `abs(display_delta) >= 4` (line 138). If a same-turn `betrayed_helper` HURT is *also* last turn's pact partner (both flags set), its `display_delta` drops from −8 to −4 under this change, shifting headline priority/wording. The spec says the cross-turn flag is "left as-is," but its *displayed magnitude* still moves. Cosmetic, but a behavior change the spec doesn't acknowledge.
- **D1 recommendation (a) puts the attacker's +4 only on the feed chip.** Even the recommended, contained option surfaces the bonus solely in `build_pd_replay_view`'s chip `display_delta`; the robot-circle animation and the two legends are separate render paths that (a) does not touch. AC5's "honest" bar is only partially met by the recommended option.
- **Win-prob models miscalibrated (acknowledged non-goal).** Retraining is explicitly out of scope; the on-disk models remain trained on the −8 distribution. No action required, but the payoff distribution shift is real.
