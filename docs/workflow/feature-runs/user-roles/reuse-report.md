## Duplication risks

These are the spots where the plan will re-create existing logic unless it forces convergence on the shared helpers.

1. **Five match-ID allocators, all non-atomic `max+1`.** `game_admin_web.py:142`, `admin_api.py:36`, `game_admin_api.py:47`, and `arena._next_match_id` (`app/engine/arena.py:90-103`, used at 167 & 216). If `match_creation.py` is added but any of these keeps scanning `Match.id` and computing `max+1` inline, the `IntegrityError`-retry (FR-4) only protects the paths that adopt it — two creates through *different* allocators can still collide on the same `M_####`. The plan must route **all five** (including arena) through the one allocator. This is the single biggest duplication trap in the feature.

2. **Four human creation sites building `Match(...)` inline.** `create_match_submit` (`game_admin_web.py:143-156`), `admin_api.create_game` (`admin_api.py:37-49`), `game_admin_api.create_game` (`game_admin_api.py:48-61`), and the new user flow. FR-3 requires every human path to record `created_by_user_id` *and* route through `match_creation.py`. If any one keeps its inline `Match(...)`, it silently creates ownerless matches and bypasses owner/cap semantics — the audit's explicit fail condition. The two arena/auto-match builders (`arena.py:169-179`, `218-228`) stay inline with `created_by_user_id = NULL` (system-created) and must **not** be folded into the human helper.

3. **Three cancel sites with the same transition.** `game_admin_cancel_match` (`game_admin_web.py:436-439`), `admin_api.cancel_game` (`admin_api.py:82-84`), `game_admin_api.cancel_game` (`game_admin_api.py:92-94`) each do `registry.stop` → `state=CANCELLED` → `cancelled_at=now` → commit behind identical `ACTIVE`/terminal pre-checks. Factor the transition once; do **not** change the existing routes' allowed-state policy or error codes (AC-7). Only the admin-dashboard template's POST target moves to the new route (FR-6) so admins can cancel `ACTIVE`.

4. **Delete cascade is order-sensitive and lives in exactly one place.** `admin_delete_match` (`admin_web.py:65-95`) has a two-pass submission sweep and `winner_player_id` nulling that exist *because* the scheduler task may write one more row during cancellation (comment at `admin_web.py:74-76`). Re-implementing it from scratch in `matches_user.py` would risk dropping the second sweep or the FK-null step. Extract verbatim into `match_deletion.py`; both the admin route and the new owner/admin route call it.

5. **`/me/matches` owner blindness.** `my_matches` (`web_player.py:397-467`) is player-join-based only. Extending it (not adding a parallel page) keeps one owner read model; a second "my created matches" page would duplicate the section-grouping logic at `web_player.py:429-462`.

## New modules justified

- **`app/engine/match_creation.py`** — consolidates the four inline `Match(...)` builders (creation-site list above) and the five `max+1` allocators into one validated, owner-recording, `IntegrityError`-retrying create. Justified because no shared creation helper exists today and FR-3/FR-4 require all human paths (plus arena's allocator) to converge here; leaving it out guarantees duplication risks 1 and 2.

- **`app/engine/match_deletion.py`** — single home for the order-sensitive delete cascade (and, per FR-6, the shared cancel state transition). Justified because the cascade currently lives only inside the admin HTML route (`admin_web.py:65-95`) and both the existing admin route and the new owner/admin route must share the exact same ordering (Risks: verbatim move). Prevents duplication risks 3 and 4.

- **`app/routes/matches_user.py`** — the new user-facing surface (`GET/POST /games/{game}/matches/new`, `POST /matches/{id}/delete`, `POST /matches/{id}/cancel`). Justified because there is no signed-in-user match-management route module today (only admin/game-admin ones); per the repo's "split human routes by audience" convention this is its own audience. It stays thin: auth (`require_user`), the per-user cap check, the owner/admin policy decision, and delegation to the two engine helpers above.
