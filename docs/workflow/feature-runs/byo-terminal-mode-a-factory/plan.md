## Architecture decisions

All decisions trace to `reuse-report.md` (no new modules; extend + one migration) and the spec FRs.

1. **Long-poll = in-handler async re-check loop (single instance).** In `next_turn` (`app/routes/agent_next_turn.py`), when no servable turn exists, loop up to a bounded window (`_LONGPOLL_WINDOW_SECONDS`, default 25): acquire a DB session, run the existing candidate-collect + `_claim_pin`, and either return the turn or **release the session** and `await asyncio.sleep(_LONGPOLL_TICK_SECONDS)` (~2s). Because it `await`s (not blocks), the FastAPI worker is free during the sleep. We are single-instance (one process, asyncio scheduler), so **no cross-process coordination is needed** — explicitly rejecting the round-2 "cross-process-safe coordination" framing as out of scope. `broadcast.py` event-wakeup is a later optimization, not v1.
   - Session lifetime (**REVISED per plan review — HIGH**): the route currently takes a request-scoped `DbSession` (from `get_session()` in `app/db.py`) that stays checked out for the ENTIRE request — so a 25s hold pins a pooled connection the whole time and exhausts the pool under moderate concurrency. Using `DbSession` "only for the first check" does NOT fix this; the checkout lives until the handler returns. **Fix: `next_turn` MUST DROP the `db: DbSession` dependency entirely** and open short-lived sessions explicitly from the async session factory, each wrapped in `async with` so it closes even on exception. Flow per tick: `async with` open session → candidate-collect + `_claim_pin` → if turn, return; else session closes → `await asyncio.sleep(tick)` → repeat. No pooled connection is held across any sleep. Cross-session safety: `_claim_pin`'s atomic conditional UPDATE still prevents double-serve even though each tick uses a disjoint session (resolves Gemini's cross-session double-pin worry). This is the riskiest change; it gets the diff-checkpoint review.
   - **Auth must not hold a session either (plan review round 2 — HIGH):** dropping `db: DbSession` from `next_turn` is NOT sufficient — `require_connection` itself depends on `get_session()`, which yields a request-scoped session held for the entire request. So the long-poll endpoint MUST NOT use ANY request-scoped session-yielding dependency: authenticate the connection inside the handler from the `X-Connection-Key` header using a short-lived `async with` session, then run the wait loop with short-lived sessions. No pooled connection is held across any sleep — from auth or from the loop.
   - Re-validate state each tick (**plan review — MEDIUM, extended round 2**): each tick MUST re-check, before claiming — connection not paused, not deleted, AND the owning user not disabled (so the account-disable guarantee holds within one tick, not lagging the full window).
2. **Turns-played counted at act-submission (exact, atomic).** In `agent_submit` (`app/routes/agent_api.py`), after a successful `submit_action` persist, increment the owning connection's `turns_played` with a SQL-level atomic update (`update(Connection).where(id==...).values(turns_played=Connection.turns_played + 1)`) — never read-modify-write (round-2 lost-update finding). Attribution (**REVISED per plan review — HIGH**): credit the increment to the **connection that authenticated the `submit_action` request** (the `sk_conn_` connection resolved by `require_connection` / `require_agent_player`), NOT `served_by_connection_id`. `require_agent_player` checks only same-user + provider-enabled, not the current pin, so any eligible live connection may legitimately submit; crediting the pin would misattribute the turn. The submitting connection is the correct, already-available owner of that call. Talk submissions do NOT increment turns.
3. **Approximate call-count folded into the heartbeat (first-class).** In `mark_seen` (`app/engine/connection_activity.py`), when the existing throttle fires the `last_seen_at` write, also `api_call_count = api_call_count + N_since_last` — i.e., increment by the throttle-window's accumulated count in the same UPDATE, or simply `+1` on the throttled write as a sampled approximation. No new per-call write. Per round-2, treat this as first-class (shipped), because polling — not turns — is the real cost proxy; dashboard labels it approximate.
4. **Storage.** New Alembic migration adds `turns_played INTEGER NOT NULL DEFAULT 0` and `api_call_count INTEGER NOT NULL DEFAULT 0` to the connections table, using the repo's `op.batch_alter_table` SQLite pattern. Columns live on the connection row; contention is low because turns_played writes are per-turn (infrequent) and api_call_count writes are throttled.
5. **Dashboard.** `web_player.py` + template surface per-connection `turns_played` (exact) and `api_call_count` (labeled approximate).
6. **Docs.** Fix `setup-mcp.md` key/header; add a Mode A section with ONE universal play-prompt that teaches the full enforced loop (poll → if phase==talk submit_talk else submit_action; save + resend `agent_turn_token` and `turn_token`) and per-client connect snippets (Claude Code, Desktop, Codex, Gemini, Cursor).

## Residual Risks (each carries a verification action — FF rule)

- **Long-poll holds a DB connection across the sleep (pool exhaustion).** verification: add an async test that runs the long-poll with no open turn and asserts the request-scoped session is closed/released during the wait (e.g., assert the engine pool checked-out count returns to baseline between ticks); must pass pre-merge.
- **Long-poll ties up a FastAPI worker.** verification: a test that fires K concurrent long-poll requests while a normal endpoint stays responsive within the window (assert the control request returns quickly); pre-merge.
- **Per-client request timeout < 25s hold (retry storms).** verification: operator 5-client matrix (Claude Code, Desktop, Codex, Gemini, Cursor) records each client's observed tolerance; if any client errors under the 25s hold, lower `_LONGPOLL_WINDOW_SECONDS` before merge. Jittered fallback hint (FR-013) mitigates synchronized retries.
- **Counter lost updates under concurrency.** verification: the increment MUST be a SQL-level atomic `+1` (not read-modify-write); a code-review check at the diff checkpoint + a test submitting multiple turns and asserting the exact count.
- **Write amplification at mark_seen.** verification: a test that issues M rapid polls within one throttle window and asserts the connection row is written at most once per throttle interval (api_call_count does not add an unthrottled write).
- **Connector regression (shared endpoint).** verification: existing connector/next-turn tests stay green; add an assertion that a connector-style immediate poll still returns without entering the long-poll hold when configured for it; Preflight Gate green.
